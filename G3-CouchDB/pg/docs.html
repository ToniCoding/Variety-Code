Guía de usuario
Replicación La replicación es un proceso unidireccional incremental que involucra dos bases de datos (una fuente y un destino). El objetivo de la replicación es que al final del proceso, todos los documentos activos en la base de datos de origen también estén en la base de datos de destino y todos los documentos que se eliminaron en la base de datos de origen también se eliminen en la base de datos de destino (si es que existieron).
Diseño de documentos CouchDB admite documentos especiales dentro de bases de datos conocidas como "documentos de diseño". Estos documentos, en su mayoría impulsados ​​por el JavaScript que escribe, se utilizan para crear índices, validar actualizaciones de documentos, formatear resultados de consultas y filtrar replicaciones.
Mejores prácticas Las mejores prácticas para el uso de CouchDB son: pre-agregar los datos, buscar alternativas al auto-incremental y no confiar plenamente en la auto-generación de UUID de CouchDB.
Guía de administración
Configuración Después de la instalación y el inicio inicial, visite Fauxton en http://127.0.0.1:5984/_utils#setup. Se le pedirá que configure CouchDB como una instancia de un solo nodo o que configure un clúster. Cuando haga clic en "Configuración de un solo nodo", se le pedirá un nombre de usuario y una contraseña de administrador. Elígelos bien y recuérdalos.
Mantenimiento La operación de compactación es una forma de reducir el uso del espacio en disco mediante la eliminación de datos antiguos y no utilizados de la base de datos o la visualización de archivos de índice. Esta operación es muy similar a la operación de vacío (SQLite ej.) disponible para otros sistemas de administración de bases de datos.

API Los componentes de la ruta URL de la API ayudan a determinar la parte del servidor CouchDB a la que se accede. El resultado es que la estructura de la solicitud de URL identifica y describe efectivamente el área de la base de datos a la que está accediendo.

Bases de datos particionadas Una base de datos particionada forma documentos en particiones lógicas utilizando una clave de partición. Todos los documentos se asignan a una partición y, por lo general, a muchos documentos se les asigna la misma clave de partición. El beneficio de las bases de datos particionadas es que los índices secundarios pueden ser significativamente más eficientes al ubicar documentos coincidentes, ya que sus entradas están contenidas dentro de su partición. Esto significa que una lectura de índice secundario dada solo escaneará un único rango de partición en lugar de tener que leer una copia de cada fragmento.
